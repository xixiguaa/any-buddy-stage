
# WorkBuddy Mini 版可研计划书

## 1. 项目背景

目标是调研 WorkBuddy 的公开功能和可能的架构思路，并据此评估一个“mini 版”本地 Agent 桌面产品的可行性。

公开信息目前只能确认到：WorkBuddy 相关产品与腾讯 CodeBuddy 体系存在关联，且外部描述中它被归类为桌面端/本地 Agent 工具，强调本地执行、办公工具接入和多智能体协作。但其内部实现细节没有可直接验证的官方技术文档，因此本可研以“公开信息 + 业界通用实现推断”为主，文中凡是架构判断都应视为假设。

## 2. 研究目标

1. 明确 WorkBuddy 可能覆盖的核心功能边界。
2. 推断其产品形态和本地 Agent 架构。
3. 识别 mini 版最小可行功能集。
4. 评估 Electron + 本地 Agent + 插件/连接器 的实现成本与风险。
5. 输出一版可落地的 MVP 方案和 4-6 周验证计划。

## 3. 已知信息

1. 外部公开描述中，WorkBuddy 被放在 CodeBuddy 体系下。
2. 产品形态偏桌面应用，强调无需复杂云端部署。
3. 典型场景包括：办公自动化、文案生成、数据处理、工具接入。
4. 公开资料中提到多智能体分工的叙事，例如规划、执行、部署类 Agent。
5. 目前能直接拿到的官方公开页面信息很少，更多细节需要后续二次核实。

## 4. 合理假设

1. 前端壳大概率是 Electron 或同类桌面框架。
2. Agent 主逻辑在本地，至少包含任务编排、上下文管理、工具调用和结果回写。
3. 云端更多承担登录、模型路由、账号、同步、审计或额度管理，而不是核心执行。
4. 本地能力会优先覆盖文件、浏览器、剪贴板、Office/IM 工具、系统命令等高频动作。
5. 真实产品会采用“可配置工具集 + 多轮任务状态机”，而不是单纯聊天窗口。

## 5. 目标用户

1. 需要频繁做办公自动化的个人用户。
2. 小团队里的运营、产品、行政、分析角色。
3. 想把 AI 用到本地文件和桌面软件里的开发者/高级用户。

## 6. mini 版功能范围

### 必做

1. 桌面聊天入口。
2. 本地文件读写与搜索。
3. 任务拆解与步骤执行。
4. 工具调用记录与可回放日志。
5. 简单工作流模板。

### 选做

1. 浏览器自动化。
2. Excel/CSV 处理。
3. IM 或邮件发送。
4. 语音输入/输出。
5. 多 Agent 协作界面。

### 不建议首版做

1. 重度云同步。
2. 复杂权限系统。
3. 大而全的插件市场。
4. 多端实时协同。

## 7. 架构假设

### 7.1 进程划分

1. Electron 主进程：窗口、系统能力、文件访问、任务调度。
2. Renderer：聊天 UI、任务面板、日志面板、配置页。
3. 本地 Agent Runtime：状态机、工具选择、上下文拼装、重试与回滚。
4. Tool 层：文件、浏览器、表格、剪贴板、系统命令、第三方连接器。
5. 模型层：本地模型或云模型的统一抽象。

### 7.2 核心数据流

用户输入 -> 任务解析 -> 计划生成 -> 工具调用 -> 本地执行 -> 结果校验 -> UI 回写 -> 归档日志。

### 7.3 推荐模块

1. `task-orchestrator`
2. `context-store`
3. `tool-registry`
4. `executor`
5. `audit-log`
6. `policy-engine`

## 8. 关键技术选型

1. Electron：桌面壳与系统集成。
2. TypeScript：主工程语言。
3. SQLite：本地状态、任务、日志。
4. 沙箱执行：限制命令和文件访问范围。
5. MCP 或自定义工具协议：统一连接外部工具。
6. 轻量向量检索：本地知识库与历史任务回忆。

## 9. 主要风险

1. 本地执行安全风险高。
2. 工具链越多，稳定性越差。
3. 任务状态不透明会让用户失去信任。
4. Electron 体积和性能需要控制。
5. 模型幻觉会在“自动办公”场景放大。

## 10. 可行性判断

### 可行

1. 做一个本地优先的 mini 版完全可行。
2. 首版可聚焦“文档 + 文件 + 表格 + 浏览器”四类能力。
3. 通过日志、确认机制和回放面板，可以把风险压住。

### 不可轻易做大

1. 全量替代办公套件不现实。
2. 深度兼容所有第三方桌面应用成本很高。
3. 多 Agent 组织化协作适合二期，不适合首版。

## 11. 建议路线

### 阶段 0：调研验证

1. 找到 WorkBuddy / CodeBuddy 的更多公开资料。
2. 归纳高频任务与典型交互。
3. 识别本地能力边界。

### 阶段 1：MVP

1. Electron 桌面壳。
2. 单 Agent 编排。
3. 文件系统工具。
4. 任务日志与回放。
5. 一个连接器：例如 CSV/Excel 或浏览器。

### 阶段 2：增强

1. 多 Agent 分工。
2. 插件系统。
3. 工作流模板。
4. 更强的审计与权限控制。

## 12. 4-6 周计划

### 第 1 周

完成资料调研、需求切片和原型草图。

### 第 2 周

搭 Electron 框架、本地存储、任务引擎骨架。

### 第 3 周

接入文件与表格工具，打通一次完整任务链路。

### 第 4 周

补日志、回放、错误处理和基础安全策略。

### 第 5-6 周

做真实办公样例测试，评估是否值得进入二期。

## 13. 结论

WorkBuddy 这类产品的核心不是聊天，而是“本地可控的任务执行系统”。如果你的目标是做 mini 版，最稳妥的路径是先做 Electron 本地 Agent 框架，再逐步加工具，不要一开始就追求全能办公自动化。
